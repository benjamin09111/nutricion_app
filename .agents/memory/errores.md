# Registro de Errores Correctores

Este archivo acumula las fallas y correcciones realizadas al agente para evitar regresiones y repetir malas prácticas. Formato estrictamente negativo.

- **Nunca crees componentes o lógicas redundantes**: NUNCA crear un componente o lógica nueva si ya existe una solución reutilizable en el codebase. Buscar exhaustivamente antes de codificar.
- **Nunca dejes persistir estados fantasma**: No dejes que el estado gestacional o variables específicas de sexo persistan si el sexo del paciente cambia a Masculino u Otro. Limpiar las variables del formulario inmediatamente.
- **Nunca uses colores fuera de la paleta definida**: Usar SOLO los códigos hexadecimales de la paleta definida en `globals.css`. Nunca usar colores genéricos de Tailwind (`green-500`, etc.). Los colores de marca son: Emerald `#8da84f` / `#74853f` / `#f7fbe8`, e Indigo `#8f70d8`.
- **Nunca ejecutes comandos en la terminal sin aprobación previa**: La terminal y comandos como instalaciones de npm/pip o migraciones manuales requieren siempre la aprobación explícita del usuario a través del flujo normal del IDE.
- **Nunca introducir caracteres UTF-8 corruptos**: Jamás escribir secuencias como `Ã¡`, `Ã©`, `Ã³`, `Ã±`, `â€"`, `â€˜`, `â€™`, `âˆž` ni ninguna variante con `Ã` en archivos fuente. El proyecto está en español (UTF-8). Guardar siempre en UTF-8 sin BOM. Después de cualquier edición, verificar que los caracteres especiales (tildes, eñes, emojis, símbolos como `—`, `×`, `∞`) se vean correctamente. Si se detecta una corrupción, corregir inmediatamente antes de continuar.
- **Nunca hardcodees URLs ni dominios en el código fuente**: Jamás escribir `localhost`, `http://localhost:3000`, `https://nutrinet.cl` ni ningún dominio directamente en código fuente (TypeScript, JSX, etc.). Usar SIEMPRE variables de entorno (`process.env.FRONTEND_URL`, `process.env.NEXT_PUBLIC_API_URL`, etc.). Las URLs deben ser configurables por ambiente (dev, staging, producción). La única excepción son los archivos `.env` y `.env.example` (que son archivos de configuración, no código fuente).
- **TODO modal debe cerrarse al hacer clic fuera (backdrop)**: El componente `Modal` tiene la prop `closeOnBackdropClick` (default `true`). Nunca crear un modal que no se cierre al hacer clic fuera, a menos que sea explícitamente requerido por UX (ej. modales de confirmación de eliminación). Si se necesita un modal que no cierre, usar `closeOnBackdropClick={false}` con justificación.
- **Nunca cargar pacientes sin filtrar por estado en selectores de formulario**: En el formulario de nueva consulta (`ConsultationFormClient.tsx`), el `SearchableSelect` de pacientes DEBE filtrar solo pacientes activos. Nunca usar `/patients?limit=100` sin `&status=Activos`. Además, implementar siempre filtro client-side defensivo (`useMemo` con `.filter(p => p.status === "Active")`) porque el caché HTTP de 5 min del endpoint `/patients` puede devolver datos stale que incluyan inactivos. La combinación server-side + client-side garantiza que nunca se muestren pacientes inactivos en selectores de creación.
