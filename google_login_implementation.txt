# Google Login Integration

Guia maestra para implementar inicio de sesion y registro con Google en un proyecto web moderno.

Objetivo: permitir que otro agente implemente el flujo completo sin romper la autenticacion existente, con un camino claro de frontend, backend, sesion y validacion.

## 1. Alcance

Este flujo cubre:

- Boton de login con Google.
- Alta o entrada del usuario con la misma accion.
- Creacion o reutilizacion de cuenta interna.
- Emision de sesion segura.
- Redireccion post-login.
- Manejo de errores y estados de carga.

No cubre:

- Password login.
- Recuperacion de contrasena.
- MFA.

## 2. Arquitectura recomendada

Separar en 4 capas:

1. UI de login.
2. Servicio de autenticacion con Google.
3. Endpoint o callback de intercambio de credenciales.
4. Capa de sesion / contexto de usuario.

Regla clave: la UI nunca debe decidir la seguridad. Solo dispara el flujo.

## 3. Flujo completo

1. El usuario entra a `/login`.
2. Hace click en `Continuar con Google`.
3. Se abre el provider de Google.
4. Google devuelve una identidad verificada.
5. El backend valida la identidad.
6. El backend crea o recupera el usuario interno.
7. El backend emite sesion.
8. El frontend actualiza el estado autenticado.
9. Se redirige al dashboard o ruta de destino.

## 4. Requisitos previos

- Proyecto con autenticacion centralizada.
- Provider Google habilitado en la consola correspondiente.
- Variables de entorno configuradas.
- Modelo de usuario interno definido.
- Ruta de retorno permitida en Google OAuth.

## 5. Variables de entorno

Define y documenta estas variables:

- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `GOOGLE_REDIRECT_URI`
- `APP_BASE_URL`
- `SESSION_SECRET` o equivalente
- `COOKIE_DOMAIN` si aplica

Si el proyecto usa Firebase, agrega tambien:

- `NEXT_PUBLIC_FIREBASE_API_KEY`
- `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN`
- `NEXT_PUBLIC_FIREBASE_PROJECT_ID`

## 6. Implementacion paso a paso

### Paso 1: Habilitar Google en el proveedor

1. Crear credenciales OAuth en Google Cloud.
2. Registrar el redirect URI correcto.
3. Activar el provider en el panel de autenticacion.
4. Verificar que los dominios permitidos incluyan el entorno local y production.

Checklist:

- El redirect URI coincide exactamente.
- El client ID y secret estan en secret manager o env vars.
- No se exponen secretos al cliente.

### Paso 2: Crear el boton de login

La vista de login debe:

- Mostrar un CTA claro.
- Bloquear clicks duplicados durante la peticion.
- Mostrar estado de carga.
- Mostrar error entendible si falla el login.

Pautas:

- No recargar la pagina manualmente.
- No mezclar logica de sesion dentro del componente visual.

### Paso 3: Implementar el servicio de Google

Centraliza la logica en un servicio o helper.

Responsabilidades:

- Iniciar el login con Google.
- Recibir el token o credential.
- Enviarlo al backend.
- Traducir errores tecnicos a mensajes de UI.

Buenas practicas:

- Un solo punto de entrada para Google auth.
- No repetir logica entre login y registro.
- No guardar tokens sensibles en localStorage si puedes evitarlo.

### Paso 4: Validar identidad en backend

El backend debe comprobar:

- Token valido.
- Firma valida.
- Audiencia correcta.
- Emisor correcto.
- Usuario no revocado.

Luego debe resolver el usuario interno:

- Si existe, actualizar ultimo acceso.
- Si no existe, crearlo.
- Si existe pero esta bloqueado, rechazar.

Campos utiles:

- `email`
- `name`
- `photoURL`
- `provider = google`
- `providerId`
- `emailVerified`
- `lastLoginAt`

### Paso 5: Crear o reutilizar cuenta

La regla mas importante es que login y registro compartan la misma identidad.

Lógica recomendada:

1. Buscar por `providerId`.
2. Si no existe, buscar por `email`.
3. Si existe por email, enlazar provider si corresponde.
4. Si no existe, crear usuario nuevo.

Evita:

- Duplicados por diferencias de mayusculas/minusculas.
- Crear una cuenta nueva si el email ya existe.
- Perder datos previos del usuario.

### Paso 6: Emitir la sesion

La sesion debe emitirse desde backend o mediante un mecanismo seguro equivalente.

Recomendado:

- Cookie httpOnly.
- Secure en production.
- SameSite acorde al dominio.
- Expiration definida.

Si el proyecto usa JWT:

- Firmar en backend.
- Renovar con rotacion segura.
- No almacenar secretos en el frontend.

### Paso 7: Sincronizar el estado en frontend

Despues del callback exitoso:

- Cargar usuario autenticado.
- Poblar contexto global.
- Redirigir a la pantalla correcta.

El contexto debe responder a:

- Loading.
- Authenticated.
- Unauthenticated.
- Error.

### Paso 8: Redireccion post-login

Usa una ruta unica de exito.

Reglas:

- Si existe `redirectTo`, respetarlo si es seguro.
- Si no, enviar al dashboard.
- Nunca permitir redirecciones arbitrarias sin validacion.

## 7. Manejo de errores

Errores comunes:

- Popup bloqueado.
- Usuario cancela el provider.
- Credencial expirada.
- Redirect URI incorrecto.
- Email no permitido.
- Usuario bloqueado.

UX recomendada:

- Mensaje corto.
- Boton reintentar.
- No mostrar stack traces al usuario final.

## 8. Seguridad

- No guardar client secret en el cliente.
- Verificar token en backend.
- Usar cookies httpOnly cuando sea posible.
- Validar email verificado si el negocio lo requiere.
- Proteger rutas privadas por sesion real, no solo por estado local.
- Sanitizar el `redirectTo`.

## 9. Estructura de archivos sugerida

Ejemplo de organizacion:

- `src/app/login/page.tsx`
- `src/components/auth/GoogleLoginButton.tsx`
- `src/services/auth/google.ts`
- `src/app/api/auth/google/callback/route.ts`
- `src/contexts/auth-context.tsx`
- `src/lib/auth/session.ts`

## 10. Checklist de implementacion

- Login page creada.
- Provider Google configurado.
- Callback/backend funcionando.
- Usuario interno creado o reutilizado.
- Sesion emitida correctamente.
- Contexto de auth actualizado.
- Redireccion post-login ok.
- Logout probado.
- Manejo de error probado.

## 11. Checklist de pruebas

Probar manualmente:

- Primer login con email nuevo.
- Login con cuenta ya existente.
- Cancelar popup.
- Redirect URI incorrecto.
- Usuario bloqueado.
- Sesion persistente al refrescar.
- Logout y regreso a login.

Probar en distintos entornos:

- Local.
- Staging.
- Production.

## 12. Errores tipicos

- Confundir login con registro: debe ser el mismo flujo.
- Crear usuarios duplicados por email.
- No validar el token en backend.
- Depender solo del estado del frontend.
- Olvidar `redirectUri` exacto.
- Mezclar logica de UI con logica de autenticacion.

## 13. Criterio de calidad

La implementacion esta correcta si:

- Un usuario puede entrar con Google en un solo intento.
- No se crean duplicados.
- La sesion persiste de forma segura.
- El usuario queda disponible para toda la app.
- Los errores son claros y recuperables.

## 14. Recomendacion para otro proyecto

Si vas a replicar este flujo en otra app:

1. Define el modelo de usuario primero.
2. Define como se guarda la sesion.
3. Configura el provider.
4. Implementa el callback.
5. Conecta el contexto global.
6. Recién despues conecta la UI.

Ese orden evita re-trabajo y rompe menos cosas.
